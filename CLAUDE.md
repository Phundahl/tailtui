# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A lightweight, keyboard-driven terminal UI — **tailTUI** (visual codename "Matrix Core") — that wraps Tailscale, built on the Charmbracelet stack. `design-spec.md` is the authoritative source for layout, interactions, and styling. The project is built **iteratively in numbered phases**, each ending with user approval before the next begins.

Branding: the product name is **tailTUI** (shown in the header; version in the footer via `appName`/`appVersion` in `view.go`). The Go module path is `github.com/Phundahl/tailtui` and the theme-override env var is `TAILTUI_THEME` (both renamed in Phase 16). The repo directory is `tailtui`.

## Commands

```bash
go run .              # run the TUI (needs a real terminal / TTY)
go build ./...        # compile everything
go vet ./...          # static checks
go test ./...         # tests (View-layer smoke tests in internal/tui/*_test.go)
go test ./internal/tui/ -run TestName -v   # run a single test
```

The View layer is testable without a TTY: construct `tui.New()`, send a `tea.WindowSizeMsg`, then call `Update`/`View` directly (see how the phase smoke tests were written). Prefer this over launching the program when verifying render/logic changes.

## Standing constraints (do not violate without being asked)

- **Live node data + live latency via the `tailscale` CLI (Phases 6–7).** Node list, details, local node, advertised routes, and the latency graph all come from the real CLI through `internal/tailscale` — mapped into the existing `internal/types` structs (never replacing them). Status from `tailscale status --json`; latency from `tailscale ping`. The **accounts** modal is live too (Phase 14): `tailscale switch --list --json`. The **Advanced Settings** modal is live too (Phase 20): local prefs read from `tailscale debug prefs` and toggled via `tailscale set --<flag>=<bool>`. The **Routing Management** modal (Phases 21–23) reads the advertised exit-node + subnet routes from the same `tailscale debug prefs` (`AdvertiseRoutes`; the exit defaults `0.0.0.0/0`+`::/0` are split out into `Prefs.AdvertiseExitNode`), lets the user **stage edits locally** (toggle exit-node, add/remove routes), and applies them via `tailscale set --advertise-exit-node/--advertise-routes` **only through the "Command Room" confirmation** (`[Enter]`), which previews the exact command and offers a clipboard copy. There is **no mock data anywhere** in the app — Phase 18 deleted the `internal/mock` package entirely: the TERMINAL_LOGS ring now **starts empty** and fills only with real runtime events, and node/latency/account data all come live from the CLI. Don't reintroduce mock data.

- **Log system (FIFO ring + `[v]` overlay, Phase 8).** `Model.logs []types.LogEntry` is the single event log; `Model.appendLog(level, message)` timestamps (`15:04:05`) and appends, enforcing a **FIFO cap of `maxLogEntries = 500`** (oldest dropped) so a long session can't leak memory. Real events are logged: exit-node action success/failure (`actionMsg`, the actual `tailscale set` stderr lands as `[ERROR]`), and status connectivity transitions (logged once on fail→ok / ok→fail, never per-tick, so a down daemon doesn't flood the ring). The bottom **TERMINAL_LOGS pane is a fixed-height tail** — `renderLogs` shows only the last 2 entries (plus the transient `fetchErr` line) so UI ticks can't bury a message. **Log lines are syntax-highlighted** (Phase 11.5) via `styles.LogLevelColor(level)` — ERROR→`Danger` (red), INFO→`Primary` (green), WARN→`Warn` (yellow), DEBUG→`Secondary`, else→`Subtle` — applied to the level chip only; the timestamp is dimmed (`Subtle`/`ModalDim`) and the message stays default text. Both renderers share the helper: `logTailLine` (over Background) and `logLine` (over Surface + bold, matching the modal status chips). `[v]` opens the full-history **log overlay** (`stateLogs`): a scrollable viewport in an opaque, Surface-filled modal framed by the **pane-style titled border** `┌─┤ TERMINAL_LOGS ├──┐` (`renderLogOverlay` + `modalTitledTop` replicate `styles.Pane`'s border opaquely, rather than the `⌐ TITLE ¬` tab the other modals use). It uses its own **`logOverlayWidth`** (~85% of the terminal, capped at 120 cols) — much wider than the standard `overlayWidth` — so timestamped log lines with network paths fit on one line instead of wrapping. `[v]`/`esc`/`q` close it; `j/k` scroll.

- **Connection toggle (`c`, `tea.ExecProcess`).** The LOCAL_NODE pane's button is **state-aware** (`renderConnectButton` / `m.localConnected()`, derived from `m.local.Conn != ConnOffline`): green `[c] Connect` when the node is down, warning-yellow `[c] Disconnect` when it's up. `[c]` runs `connectCmd(!localConnected())` → `tailscale up` (down→up) or `tailscale down` (up→down) via `tea.ExecProcess` so an auth URL printed by `up` is visible and interactive — **no sudo** (the user is already a configured operator). `connectDoneMsg` logs the outcome and fires `fetchStatusCmd` so the new state shows immediately. The footer mirrors this: a dynamic `[c] Connect`/`[c] Disconnect` hint and a `● CONNECTED`/`○ DISCONNECTED` status that track `m.local.Conn`.

- **Operator setup (`O`, `tea.ExecProcess`).** Tailscale's `checkprefs access denied` happens when the user isn't the tailscaled operator. `[O]` (shift+o) runs `operatorSetupCmd`: a `tea.ExecProcess` that **suspends the TUI, releases the terminal**, and runs interactive `sudo tailscale set --operator=$USER` (`currentUser()` prefers `$USER`, falls back to `os/user`). stdin/stdout/stderr are left unset so ExecProcess wires them to the program's terminal (the sudo password prompt works); the TUI is restored automatically on exit and an `operatorDoneMsg` logs the result and triggers an immediate `fetchStatusCmd` so the new permissions take effect. This is the one place we hand the real terminal to a child process.

- **Exit Latency (LOCAL_NODE pane).** The local node has no self-RTT, so its dashboard latency field is relabeled **"Exit Latency:"** and shows the live ping to the *active exit node* (the node all traffic routes through): `renderExitLatency` reads `Model.latency[activeExitNodeIP()]`. To keep it live even when the exit node isn't highlighted, the ping ticker pings **both** the selected node **and** the active exit node each tick. Shows `N/A` when no exit node is active (or it's offline), `—` until the first sample arrives.
- **Exit State (LOCAL_NODE pane, Phase 17).** The LOCAL_NODE field formerly labeled **"State:"** (which showed the *local* node's own connectivity) is renamed **"Exit State:"** and is now strictly tied to the exit-node feature: `renderExitState` scans the peer list (the source of truth) for the `IsActiveExitNode` peer and renders **that peer's** route status — `connSymbol` + `connText` → `DIRECT`/`RELAY (via …)` — exactly as it comes from `tailscale status --json`. When no exit node is active it shows a dim **`N/A`**, keeping the LOCAL_NODE layout static and consistent with the sibling `Exit:` / `Exit Latency:` inactive readouts. The local node's *own* `Conn` still drives the `[c]` connect button and the footer status, just not this field.

- **LOCAL_NODE action buttons & flush symmetry (Phases 20–21.1).** The LOCAL_NODE pane stacks the state-aware `[c]` connect button over a single grouped action row `[S] Advanced Settings    [R] Routing` (`renderActionButtons`). The pane height is `localNodeH = 11` and **PEER DETAILS is locked to it (`detailsH = localH`)** — so any change to the LOCAL_NODE content height must keep the two heights equal (the horizontal grid line under the top panes must stay unbroken). `[S]`/`[R]` are **uppercase** (shift); the lowercase `s`/`r` are explicit no-ops in `Update`, reserved for the future SSH- and routing-related actions. Don't bind lowercase `s`/`r` without freeing them deliberately.

- **Advanced Settings modal (`S`) — optimistic + reconcile.** Toggling a preference (`Space`) flips the local `Model.prefs` field **immediately** for instant feedback **and** fires `setPrefCmd` → `tailscale.SetPref` (`tailscale set --<flag>=<bool>`) off the UI thread; the resulting `prefActionMsg` logs the outcome and **re-fetches prefs** (`fetchPrefsCmd`) so the checkbox reconciles with the daemon's true state, reverting a failed flip. Never block the Update goroutine on the CLI; never hardcode the flag list outside `settingDefs`.

- **Routing Management modal (`R`) — local working copy, applied only via the Command Room.** Edits operate on a **working copy** (`Model.routingExitNode` / `routingRoutes`) snapshotted from `prefs` at open; they **must never mutate `m.prefs`** (the daemon's last-known truth). Added routes are **validated with `net.ParseCIDR`** before entering the list; invalid input is rejected (red-flashed, cleared) without crashing. A `[d]`-deleted route is stashed in `lastDeletedRoute` and **pre-filled into the next `[a]`** (a lightweight pseudo-undo / edit-typo affordance) — the pre-fill flows through the same `net.ParseCIDR` validation, so it is not a bypass. The CIDR editor is a `bubbles/textinput` (`routingInput`), styled to blend into the modal **Surface**. Two things keep it from rendering a black block (the "black box" glitch): (1) **no in-field `Placeholder`** — a placeholder makes the empty field render via `placeholderView`, which fills the remaining Width with **raw, unstyled (near-black) spaces** that no outer Surface wrap can recolor (they sit mid-line after a reset); with no placeholder the main path pads with `TextStyle` (Surface bg) instead, so every cell is Surface (the prompt label already shows the "(e.g., 192.168.1.0/24)" example). (2) the cursor is a **visible bright Primary block**: `bubbles/cursor` draws its visible cell with `Style.Reverse(true)`, which swaps fg/bg at display time, so the **displayed background is `Style`'s Foreground** — set `Cursor.Style` = `Foreground(Primary).Background(Bg)` so the reverse yields a solid Primary block (dark glyph), neither invisible (the 23.2 `Foreground(Surface)` camouflage, which blended into the modal) nor a black block. `Cursor.TextStyle = ModalText` keeps the blink-"off" phase as normal text on Surface. (`PromptStyle`/`TextStyle` also paint Surface.) While in input mode an early guard in `updateOverlay` routes **every** key (including `esc`/`q`) to `updateRoutingInput`, so the global close handler can't fire mid-entry (`Esc` cancels the entry; `Enter` confirms). `routingDirty` prevents a late `prefsMsg` from clobbering staged edits; `refreshRoutingOverlay` re-renders **and** resizes the viewport when switching sub-modes so the editor is never clipped.

- **Routing execution — the "Command Room" (`stateRoutingConfirm`, Phase 23).** The **only** place routing changes reach the daemon. `Enter` in the routing list opens a floating confirmation overlay (`renderRoutingConfirmOverlay`, rendered directly from model state like the settings modal) showing the **exact command** to be run — assembled by `tailscale.AdvertiseCommandString(exitNode, routes)` and identical to what executes — plus the **Admin Console approval reminder**. The command **always sets both flags** (`--advertise-exit-node=<bool>` and `--advertise-routes=<csv>`); an empty routes list renders `--advertise-routes=` (the idiomatic "clear", never omit). Long previews wrap safely (`wrapCommand` = word-wrap + hard-wrap fallback for the comma-joined CIDR token). Keys: **`Enter`** applies — fires `setRoutingCmd` → `tailscale.SetRouting` **off the UI thread**, closes the modal to `stateMain`, and the `routingActionMsg` logs the executed command + batches `fetchStatusCmd`+`fetchPrefsCmd`; **`c`/`C`** copies the command to the system clipboard via `copyRoutingCmd` (async, **`github.com/atotto/clipboard`** — shells out to pbcopy/wl-copy/xclip/clip, so a missing tool logs an error instead of crashing) and flashes a `✓ Copied!` indicator (`routingCopied`, set by `clipboardMsg`), staying in the modal; **`Esc`/`q`** go **back to the routing list** (not all the way out) without applying. `updateOverlay` routes `stateRoutingConfirm` keys to `updateRoutingConfirm` *before* the global esc/q close handler so Esc means "back," not "quit the feature."
- **Theme Engine (no more strict-ANSI rule).** Colors are TrueColor hex, centralized in a `styles.Theme` struct (`internal/styles/theme.go`). `DefaultTheme()` is the **"Matrix Core"** master design — the EXACT hex codes from the style guide's YAML frontmatter (`_designs/00_STYLE_GUIDE.md`): `primary #6bfb9a`, `background #0e150f`, `surface-container #1a211b` (Surface), `surface-bright #333b34`, `outline-variant #3d4a3e` (borders), `on-surface #dde5da` (text), `outline #869486` (dim), `tertiary #ffdd75` (warning), `error #ffb4ab`. `LoadTheme()` reads the **native Omarchy** theme from `~/.config/omarchy/current/theme/colors.toml` — a flat TOML table of hex strings (`accent`, `foreground`, `background`, `color0`–`color15`) parsed with `github.com/pelletier/go-toml/v2`. Path overridable via `TAILTUI_THEME`. Mapping is per-field, so any missing key keeps its default and a missing/malformed file falls back entirely — never crashes or leaves blanks. Omarchy→Theme: `accent`→PrimaryAccent, `color2`→SecondaryAccent, `background`→Background, `color0`→Surface, `color8`→SurfaceBright/BorderInactive/TextDim, `foreground`→TextNormal, `color3`→Warning, `color1`→Error. `main` calls `styles.Apply(styles.LoadTheme())` at startup. `Apply` rebuilds the package color vars (`Primary`, `Secondary`, `Subtle`, `Warn`, `Danger`, `Fg`, `Bg`, `Surface`, `SurfaceBright`, `BorderInactive`) and derived styles; helper funcs (`Pane`, `Divider`, `LatencyGraph`, …) read them at call time. Add new colors as `Theme` fields, not raw codes. Hex degrades to ANSI on non-TrueColor terminals.
- **Elm architecture.** Standard Bubble Tea Model/Update/View; keep them split across `internal/tui/{model,update,view}.go`.

## Architecture

```
main.go                     entry point: tea.NewProgram(tui.New(), WithAltScreen)
internal/
  types/types.go            domain models (Peer, LocalStatus, enums) — CLI-agnostic
  tailscale/tailscale.go    live adapter: status/ping/set/switch/login/logout → types structs
  styles/theme.go           Theme struct, DefaultTheme (Matrix Core), Omarchy TOML loader
  styles/styles.go          theme-derived styles (Apply), Divider/Bar/LatencyGraph helpers
  styles/pane.go            Pane(): sharp single-line box with title in the top border
  tui/
    model.go                Model struct, New(), Init(), selectedPeer()
    update.go               Update(): resize + quit; everything else delegated to the list
    peerlist.go             bubbles/list construction, custom 1-line delegate, sort logic
    overlay.go              help/routes/settings modal state machine + floating-modal compositor (x/ansi)
    settings.go             Advanced Settings modal: settingDefs (5 toggles → tailscale set flags), openSettings
    routing.go              Routing Management modal: openRouting + working copy, routingBody, list/input/confirm handlers, CIDR textinput, Command Room (tailscale set + clipboard)
    poll.go                 async cmds: status fetch + refresh tick, ping + ping tick, exit-node action
    view.go                 View(): computeLayout() + pane rendering + routesSummary
```

Key design points that span multiple files:

- **`types` is decoupled from any Tailscale wire format on purpose.** `Peer`/`LocalStatus` use semantic enums (`ConnType`, `NodeType`, `OS`) so the view picks glyphs/colors by switching on a type, never by string-matching CLI output. The `internal/tailscale` adapter maps wire data → these structs.

- **The Tailscale adapter (`internal/tailscale`) is the only place that knows the CLI/JSON shape.** Private wire structs (`status`/`node`/`user`) unmarshal a *subset* of `tailscale status --json`; only `types.LocalStatus`/`types.Peer` cross the package boundary, so nothing downstream sees a Tailscale-specific field. Mapping decisions: `ConnType` from `Online`+`CurAddr` (online & has current endpoint → DIRECT, online & no endpoint → RELAY via `Relay` DERP region, else OFFLINE); `NodeType` = exit (if `ExitNodeOption`) → subnet-router (if it has non-default `PrimaryRoutes`) → regular; `OffersExitNode`=`ExitNodeOption`, `IsActiveExitNode`=`ExitNode`; `AdvertisedRoutes` = `PrimaryRoutes` minus the exit defaults (`0.0.0.0/0`, `::/0`); local `User` looked up via `Self.UserID` in the `User` map; local `LocalIP` = first RFC1918 host in `Self.Addrs` (the LAN IP isn't a top-level field); `LastSeen` humanized to a relative label ("Connected"/"3h ago"). **Peers are sorted by hostname in the adapter** before returning — map iteration is random, so without this the list would reshuffle every poll. Note: `tailscale status` carries **no per-peer Version**, so the `Peer.Version` field (and the PEER DETAILS `Version:` line) was **removed entirely in Phase 17** rather than displayed empty. Errors unwrap `*exec.ExitError` to surface the CLI's stderr.

- **Async, non-blocking data (`poll.go`).** `os/exec` must never run on the Update goroutine. `fetchStatusCmd()` is a `tea.Cmd` whose closure (run by Bubble Tea in its own goroutine) shells out under a `context.WithTimeout(refreshInterval-1s)` and returns a `statusMsg{local, peers, err}`. `Init()` returns `tea.Batch(fetchStatusCmd(), tickCmd())` — fetch once immediately, then `tickCmd()` (`tea.Tick`, `refreshInterval = 4s`) re-fires a fetch + reschedules itself every cycle. `Update` handles `statusMsg` via `applyStatus` and `tickMsg` by batching the next fetch+tick.

- **`applyStatus` folds a poll into the model without disrupting the user.** It stores `fetchErr` (kept on error, last good data stays on screen — never blanks), refreshes `m.local`, and rebuilds the list via `peers.SetItems(peerItems(...))`. It **preserves the highlighted node by hostname** (re-`Select`s it after the rebuild) so a 4s refresh never yanks your selection, and **skips the peer rebuild while `FilterState()==Filtering`** so a poll can't interrupt active `/` typing (the next tick reconciles). A fetch error renders as a red `> [ERROR] …` line atop the TERMINAL_LOGS pane.

- **Live latency (`tailscale ping`, `poll.go`).** A second ticker (`pingTickCmd`, `pingInterval = 2s`, started in `Init`) pings **only the highlighted online node** — `pingCmd(ip)` runs `tailscale ping --c 1 <ip>` off-thread and `tailscale.Ping` parses the `… in 137ms` tail (regex) into an int, returning an error (not a zero) when the node doesn't answer. `applyPing` appends the sample to `Model.latency` (a `map[ip][]int`, capped at `maxLatencySamples = 40`), drops failed pings, and updates that node's list item in place so the LATENCY HISTORY pane (and its MIN/AVG/MAX) refresh immediately. `applyStatus` re-injects `Model.latency` into the fetched peers via `withLatency`, so accumulated samples survive a 4s status refresh instead of resetting. History is keyed by Tailscale IP and accumulates per node, so navigating away and back keeps a node's graph. The **local node has no latency** — `tailscale ping` to self returns "is local Tailscale IP" (no RTT), so the LOCAL_NODE inline sparkline reads empty; the live graph is always the selected *peer's*. The adapter no longer synthesizes latency.

- **Exit-node toggle (`x`) is wired to the daemon, optimistically.** `toggleExitNode` still rewrites the list items immediately (instant feedback) **and** issues `setExitNodeCmd` → `tailscale.SetExitNode` (`tailscale set --exit-node=<ip>`, empty string clears). The optimistic flags hold until the next status poll reconciles the model with the daemon's `peer.ExitNode` truth: a successful change shows no flicker (optimistic == daemon), a failed one reverts on the next poll and its error (`actionMsg.err`) surfaces in the logs pane. Only `x` maps to this action (the former `t` alias was removed). The footer `[x] Exit Node` hint shows **only when the highlighted peer offers exit-node service**. (The design-spec `p` ping-as-action is deferred — intentionally absent from the footer and help until implemented. The Advanced Settings modal is bound to uppercase **`S`**; lowercase `s` is explicitly reserved/ignored for the future SSH-as-action.)

- **Intelligent peer sort (adapter-authoritative).** `mapPeers` sorts into four buckets — exit nodes (0), subnet routers (1), online regular peers (2), offline regular peers (3) — each alphabetical by hostname (`sortBucket`); offline nodes are pinned to the bottom. This replaced the earlier flat alphabetical sort. The tui's `sortPeers`/`nodeRank` stable pass is now redundant-but-order-preserving (specials are already first), kept only as a guard for any non-adapter item source.

- **Live routes modal.** `routesBody` renders the peer's real `AdvertisedRoutes` (its daemon `PrimaryRoutes`) with the gateway = the node's Tailscale IP, LATENCY = the node's live ping reading (`—` until a sample arrives), and a `[ ROUTING ]` (`StatusOK`) chip for every approved route. The old index-based mock columns (`routeLatency`/`routeStatus`, with fake DISABLED/CONFLICT/ERR variety) are gone. Styling/layout of the table is unchanged.
- **The peer list drives the details pane.** The middle pane is a `bubbles/list.Model`; `Model.selectedPeer()` reads `list.SelectedItem()`, and `View` renders that peer on the right every frame. There is no separate "selected index" state to keep in sync — the list is the single source of truth for selection.
- **`peerDelegate` renders one dense line per node** (`> 󰖟 [EXIT] amsterdam-exit ●`) instead of the default two-line delegate, to match the spec's high-density list. Exit nodes and subnet routers are stable-sorted to the top via `nodeRank`.
- **Custom search/filter (`search.go`, Phase 13) — NOT bubbles/list's built-in.** `list.SetFilteringEnabled(false)`; we own filtering for explicit fzf-style modes and panic-proof clamping. State: `Model.allPeers` (full sorted set, source of truth), `searchQuery`, `searchFocused`. Two modes: **Input** (`searchFocused`, box focused) and **Normal** (blurred). `Update` routes ALL keys to `updateSearchInput` while `searchFocused`, so nothing leaks to the command switch or the list keymap while typing:
  - **Input Mode:** printable runes type (incl. `j`/`k`); `↑`/`↓` and `Ctrl+j`/`Ctrl+k` navigate the filtered list (via `searchNav` → `selectClamped`); `backspace` edits; **`Enter`/`Esc` blur → Normal Mode, keeping the filter applied**.
  - **Normal Mode:** `/` (re)enters Input Mode; **`Esc` clears the filter** (full list) when one is applied; `j`/`k`/arrows navigate (`wrapNav` + list).
  Matching is case-insensitive fuzzy *subsequence* over `Peer.FilterValue()` (hostname+tags), preserving the special-node-first order. The search box is the NODES pane **title**: `SEARCH: <q>▌` in Input Mode (trailing block cursor) vs `FILTER: <q>` when blurred (no cursor) vs `FILTER NODES...` when inactive — so the cursor presence *is* the mode indicator.
- **Cursor clamping = crash fix (Phase 13).** Every cursor move funnels through `selectClamped(i)` (clamps to `[0, len(items)-1]`, safe on empty); `applyFilter` rebuilds the visible items and **resets the cursor to 0** so a shrinking match set can't leave a stale index, and `applyStatus` re-clamps after a refresh. This eliminates the out-of-bounds panic seen when filtering a long list (cursor far down) to a few items and then navigating.
- **Wrap-around navigation.** `wrapNav` intercepts the single-step nav keys (`j`/`k`, ↓/↑) in Normal Mode and jumps to the opposite end *only* when already at a boundary (top→bottom, bottom→top); otherwise it returns `handled=false` and the key falls through to the list so normal movement and pagination are untouched. Boundaries are computed over `list.VisibleItems()` (== the current filtered items) using `Index()`/`Select()`. Filtering is now synchronous (our own), so this path is fully exercisable by calling `Update` in isolation (unlike the old async bubbles filter).
- **Exit node state lives in the list items, not in `LocalStatus`.** `Peer.IsActiveExitNode` is the single source of truth; at most one peer has it set. `Peer.OffersExitNode` is a *capability* flag — `toggleExitNode` (`x`) is a no-op unless the highlighted peer offers exit-node service. When it does toggle, it rewrites every peer via `list.SetItem` — enabling the highlighted one and clearing the rest, or clearing all if it was already active. The dashboard's `Exit:` value is *derived* via `Model.activeExitNodeName()` (scans the list), so it stays in sync automatically; `LocalStatus.ExitNode` is left empty on purpose. The active node is marked in the list with a soft yellow `󰖟 exit` text label (`styles.ExitName`) — a minimal marker, not a filled chip.
- **Advertised routes.** `Peer.AdvertisedRoutes` holds subnet CIDRs (subnet routers only). When the selected peer has routes, `renderDetails` appends a yellow `[e] N advertised routes` hint line (`styles.Caution`). The PEER DETAILS pane is sized to `localNodeH` (≥ 8 inner rows after Phase 18's tightening), which comfortably fits the 6 identity rows (IDENTITY · OS · IP · Conn · Tags · Last Seen — the `Version:` row was dropped in Phase 17) + the hint, so it's never clipped by the pane's `MaxHeight`; when there's no hint the Pane just pads the bottom. Pressing `e` on such a peer opens the full routes overlay. (Clipped after the Phase 9 reshuffle, restored in Phase 10, and made symmetric — fixed height — in Phase 11.)
- **Advanced Settings modal (`S`, Phase 20 / rebound Phase 20.1).** A master/detail overlay (`stateSettings`) for toggling five live local-node preferences. Opened with **uppercase `S` (shift+s)** — lowercase `s` is explicitly ignored in `Update` (a no-op `case "s"`) to keep it free for the future SSH-as-action. The LOCAL_NODE pane gains a second stacked action label **`[S] Advanced Settings`** (`renderSettingsButton`) under the `[c]` connect button; to keep the top panes flush, `localNodeH` went **10 → 11** (and `detailsH = localH` follows automatically, so PEER DETAILS grows by the same row — the dropped/extra space is absorbed by its Pane padding). The modal is **rendered directly from model state** (`m.prefs` / `m.settingCursor`) each frame via `renderSettingsOverlay` — **not** the shared viewport — so toggling re-renders instantly. It composites two opaque, Surface-filled side-by-side `modalBox`es over the still-visible base (`overlayCenter`): **left** `ADVANCED_SETTINGS` (focused → Primary border) lists the five toggles with a `❯` cursor and a colored checkbox (`[x]` green `StatusOK` when on / `[ ]` dim when off); **right** `DESCRIPTION` (dim `BorderInactive` border) shows the highlighted toggle's help text (`wrapText`) and the exact `tailscale set --<flag>=<next>` command. The five toggles map to `tailscale set` flags via `settingDefs` (`settings.go`, data-driven get/set closures over `types.Prefs`): Accept Subnet Routes (`--accept-routes`), Allow LAN Access (`--exit-node-allow-lan-access`), Run Tailscale SSH (`--ssh`), Accept MagicDNS (`--accept-dns`), Shields Up (`--shields-up`). **State is read live** from `tailscale.GetPrefs` (`tailscale debug prefs` → JSON `ipn.Prefs` subset: `RouteAll`/`ExitNodeAllowLANAccess`/`CorpDNS`/`RunSSH`/`ShieldsUp` → `types.Prefs`), fetched in `Init` and on every `[S]` open (`fetchPrefsCmd` → `prefsMsg`). Keys (consumed entirely by the modal): **`j/k`** move `settingCursor` (clamped); **`Space`** flips the highlighted pref **optimistically** and fires `setPrefCmd` → `tailscale.SetPref` (`tailscale set --<flag>=<bool>`) off the UI thread; **`Esc`/`q`** close. The `prefActionMsg` logs the outcome and **re-fetches prefs** to reconcile the checkbox with the daemon (reverting a failed optimistic flip), mirroring the exit-node reconciliation pattern. `modalTitledTopColor` generalizes the opaque `┌─┤ TITLE ├─┐` top border to a caller-chosen focus color (the existing `modalTitledTop` now delegates to it with `Primary`); `modalBox` clips each content line (`MaxWidth`) so an over-long flag can never wrap and break the flush overlay.

- **Routing Management modal (`R`, Phases 21–23).** A floating overlay (`stateRouting`) surfacing — and locally editing — this node's advertised exit-node + subnet routes (applied via the Command Room, `stateRoutingConfirm` — see the execution constraint above), **styled after ACCOUNT_MANAGEMENT** (it reuses the same viewport-modal chrome: `⌐ ROUTING_MANAGEMENT ¬` tab title, top/bottom `modalDivider`s, and the two-column `gridLine`/`accountKey` keymap). The LOCAL_NODE pane's action label **`[R] Routing`** sits on the **same horizontal row** as `[S] Advanced Settings` (`renderActionButtons` centers `[S] Advanced Settings    [R] Routing` on one line, below the `[c]` connect button) — grouped horizontally in Phase 21.1 so the pane isn't bottom-heavy, which also let `localNodeH` revert to **11** (it stays locked to `detailsH = localH`, so PEER DETAILS keeps the same height and the top borders stay flush). Opened with **uppercase `R` (shift+r)** — lowercase `r` is explicitly ignored in `Update` (a no-op `case "r"`) and reserved for a future routing action. `routing.go` owns it: `openRouting` **snapshots** `prefs.AdvertiseExitNode`/`AdvertiseRoutes` into an **editable working copy** (`Model.routingExitNode` / `routingRoutes`) and builds the shared viewport from `routingBody`; the modal renders the **fixed Exit Node (Advertise) toggle** (`[ON]`/`[OFF]`) at cursor index 0, then each **advertised subnet route** (cursor 1..N) — or a dim **`No custom routes advertised.`** placeholder. The **highlighted row uses the inverted `styles.AccountActive` bar** (green bg, dark text), exactly like the accounts modal's active session.

  **Phase 22 — local editing (no CLI execution).** All edits mutate the working copy only — `m.prefs` (the daemon's last-known truth) is never touched; the diff is applied to `tailscale set` in Phase 23. The modal has two sub-modes:
  - **List mode** keys: **`j/k`** navigate (`routingCursor`, clamped to `routingItemCount()` = 1 + len `routingRoutes`); **`Space`** toggles `routingExitNode` (only on item 0); **`d`** removes the highlighted route (clamps the cursor when the last item goes); **`a`** enters input mode; **`Esc`/`q`** close. The keymap shows `[J/K] NAVIGATE · [A] ADD ROUTE · [SPACE] TOGGLE · [D] REMOVE · [ESC] CLOSE`.
  - **Input mode** (`routingInputMode`) embeds a **`bubbles/textinput`** (`routingInput`, themed to the modal surface) with the prompt `Enter CIDR (e.g., 192.168.1.0/24):`, replacing the keymap with `[ENTER] CONFIRM · [ESC] CANCEL`. **`Enter`** validates via `net.ParseCIDR` — valid → append to `routingRoutes`, exit input mode, highlight the new route; invalid → set `routingInputErr` (red prompt flash) and clear, staying in input mode (no crash). **`Esc`** cancels back to list mode. Input mode owns **all** keys (including `esc`/`q`) via an early guard in `updateOverlay` → `updateRoutingInput`, so the global close handler can't fire mid-entry.

  Any edit sets `routingDirty`; a late `prefsMsg` re-snapshots the working copy from fresh prefs **only while clean and not mid-entry**, so it never clobbers edits. `refreshRoutingOverlay` re-renders **and resizes** the viewport (the body height changes between sub-modes, so a stale height would clip the editor). Data is read live from `prefs` (`fetchPrefsCmd` on `Init` and every `[R]` open).

- **Overlay state machine.** `Model.state` (`stateMain`/`stateHelp`/`stateRoutes`/`stateAccounts`/`stateLogs`/`stateSettings`/`stateRouting`/`stateRoutingConfirm`) gates input and rendering. `internal/tui/overlay.go` owns it: `openHelp`/`openRoutes`/`openAccounts`/`openLogs`/`openRouting` size a shared `bubbles/viewport` and set its content (the **Advanced Settings** modal, `openSettings`, is the exception — it renders **directly from model state** each frame via `renderSettingsOverlay`, not the viewport). `updateOverlay` handles keys while an overlay is open — `esc`/`q` close any overlay (`?` also closes help, `v` also closes the log overlay). The viewport-modal footer hint (`renderOverlay`) shows the **`-- KEYBOARD INPUT MODE ACTIVE --`** banner **only** in a genuine inline text-input state (the routing CIDR editor, `routingInputMode`); read-only / navigation overlays (Help, Accounts, the routing list) show a plain `[Esc] Close` hint instead (Phase 23.1). Help/routes/logs **scroll** the viewport; the **accounts, settings, and routing modals consume their own keys** (navigation + actions), nothing falling through. The isolation rule lives in `Update`: when `state != stateMain`, keys route to `updateOverlay` and never reach the peer list. **Exception:** while the routing modal is in CIDR **input mode**, an early guard hands *every* key (including `esc`/`q`) to `updateRoutingInput` so the close handler can't fire mid-entry. Overlays re-size on `WindowSizeMsg` via `resizeOverlay`. The accounts modal (`l`, `accountsBody`) shows a green active-session bar + inactive rows + a two-column action grid, backed by **live** profile data; the settings (`S`) and routing (`R`) modals are detailed in the constraints above.

- **Live account management (Phase 14 + sudo coverage 24.2 / 24.3).** Profiles come from `tailscale.Accounts` (`tailscale switch --list --json` → `[]types.Account{ID, Email, Active}`, active sorted first). Fetched in `Init`, on `[l]` open, and after every account action (`fetchAccountsCmd` → `accountsMsg` on success, `accountsLockedMsg` on the profile-lock error — see below — which re-`SetContent`s the modal if open). On Linux the profile store is root-owned, so **all four mutating actions** run as `tea.ExecProcess` invocations wrapped in `sudo` (closing the modal — `state = stateMain` — *before* dispatch so the post-`ExecProcess` repaint lands on the dashboard, not a stale overlay): **`Enter`** switches to the highlighted *inactive* profile (`switchAccountCmd` → `sudo tailscale switch <id>`); **`a`** adds an account (`addAccountCmd` → `sudo tailscale login` — interactive for both the sudo password and the auth URL); **`d`** removes the highlighted *inactive* profile (`removeAccountCmd` → `sudo tailscale switch remove <id>`); **`l`** logs out (`logoutCmd` → `sudo tailscale logout`). Active-profile guards: `Enter`/`d` on the active row are no-ops. All four return `accountActionMsg{desc, err}`, which logs the outcome and batches `fetchAccountsCmd` + `fetchStatusCmd` + `fetchPrefsCmd` on success (login can change which prefs the daemon will return for this user) and `fetchAccountsCmd` on error. The orphaned `tailscale.SwitchAccount` / `RemoveAccount` / `Logout` adapter functions were deleted across 24.2 / 24.3; only the `Accounts` reader remains. Styling/dimensions of the modal are unchanged — only the data flow and key routing.

- **Profile-store lock detection (Phase 24.3).** `fetchAccountsCmd` is re-fired on every refresh and account action, so an unprivileged session would flood the log ring with the recurring `Access denied: profiles access denied` line. The guard: `isProfilesAccessDenied(err)` (case-insensitive substring match on "profiles access denied" / "access denied") classifies the error and the command returns a **distinct `accountsLockedMsg{}`** instead of a generic `accountsMsg{err: …}`. The Update handler for `accountsLockedMsg` sets `Model.profilesLocked = true`, clears `m.accounts` (no stale list from a prior elevated run lingering behind the hint), resets the cursor, refreshes the modal if open — and **logs nothing** (the lock flag is the only side-effect). A successful `accountsMsg` clears the flag, so freshly-granted perms drop the hint on the next refresh. `accountsBody` substitutes the empty-state line for the wrapped (`wrapText`, modal-width aware) hint **"Profile store locked. Run tailTUI with sudo to view and manage accounts."** when the flag is set; the divider + two-column action grid stay in place (single-text substitution, no layout change), and `[a]` still works since it pre-elevates via sudo itself.
- **Floating modals (compositing).** `View` always renders the full base layout first, then `renderOverlay(base)` blits the modal on top so the dashboard/list/details stay visible behind it. lipgloss v1 has no overlay primitive, so `overlayCenter` does it by hand: for each background row covered by the modal it keeps the left columns (`ansi.Truncate`), splices in the modal line, then resumes the background past the modal (`ansi.TruncateLeft`) — both are ANSI-aware and carry SGR state across the cut, with explicit `\x1b[0m` resets isolating the three segments. This is why `internal/tui` depends directly on `github.com/charmbracelet/x/ansi`.
- **Brutalist sharp panes (`styles.Pane`).** Every logical region is a `styles.Pane(title, body, w, h, focused)` — a SHARP single-line box (`┌─┐│└┘`, `lipgloss.NormalBorder` geometry built by hand) with the title embedded in the top border: `┌─┤ LOCAL_NODE ├────┐`. No rounded corners anywhere (Matrix Core is strictly sharp/brutalist). `Pane` renders to exact OUTER w×h (border + `Padding(0,1)` included); use `styles.ContentWidth(outer)` for the usable inner width. The title is left-aligned in the top border to match the mockups (the spec text says "centered" but the images are left).
- **Layout is centralized in `computeLayout` (Phase 11 grid).** Both `Update` (which sizes the list via `SetSize`) and `View` (which draws) read the same geometry via `m.layout()`. The screen is: header bar · a body band · footer bar. The body is two columns joined by a 1-col gutter, **each summing to `bodyH` so they join flush** (`JoinHorizontal(Top, …)`): **left = LOCAL_NODE (fixed `localNodeH`) over the FILTER NODES list (flex)**; **right = PEER DETAILS (fixed) over LATENCY HISTORY (flex) over the TERMINAL_LOGS tail (fixed `logsHeight`)**. Each column's flex pane is computed as the remainder, so the sums are exact. **PEER DETAILS is LOCKED to the LOCAL_NODE height (`detailsH = localH`)** for horizontal symmetry — the bottom border under the two top panes aligns across the screen regardless of selection, and short content (or the missing routes hint) is absorbed by the Pane's own bottom padding. When the terminal is short, `localH` (and thus `detailsH`) is clamped toward `localFloor` so the flex node list keeps `nodesFloor` and LATENCY keeps `minLatencyH`. The list's pane height (`nodesH`) depends only on `localH`+`bodyH`, **not** on the selected peer, so navigation never re-sizes the list — only `WindowSizeMsg` does. `WindowSizeMsg` is intercepted (not forwarded to the list) so the list gets the *pane* size.

- **Flush-layout invariants (Phase 9 border fix).** The full render must be EXACTLY `width × height`; any single line wider than the terminal wraps and pushes the layout down, scrolling the top borders off the alt-screen (the reported regression). Two guards enforce this: (1) `styles.Pane` clips its content with `MaxHeight`/`MaxWidth` (lipgloss `Height`/`Width` only pad, never truncate — an over-tall body, e.g. a wrapped log line, otherwise grows the pane); (2) `styles.Bar` (header/footer) clips the left segment and caps the whole line at `width`, so a long hint string can't overflow. Keep both when touching panes or bars.
- **Border = focus, color = state.** Only the focused pane (the peer list / NODES) gets the bright `Primary` border; all others use the dim `BorderInactive`. The selected list row is a full-width `SurfaceBright` (`#333b34`) highlight bar with a leading `❯` pointer (`styles.Selected`, built from plain text so the bar paints uniformly). Content colors follow state: accent = online/direct/routes, warning (`Caution`, yellow) = exit node/relayed, error = conflict, dim = offline/secondary. `●`/`○` mark online/offline.
- **Latency graph.** The LATENCY HISTORY pane uses `styles.LatencyGraphArea(values, width, height)` — a **multi-row** vertical bar chart that fills the (now flexible) pane: bar *height* is scaled to the series min/max using eighth-block glyphs for sub-row precision, while bar *color/weight* encodes absolute latency via `latencyColor` — faint accent <30ms, solid accent 30–59, bold warning 60–99, bold error ≥100ms. `renderLatency` sizes the chart to the pane's remaining inner height (`latencyH − stats − spacer`); it falls back to the single-row `LatencyGraph`/`LatencyGraphWidth` at height 1. The compact dashboard "Exit Latency" readout keeps the single-color `styles.Sparkline`. All pull colors from the active theme (`Primary`/`Warn`/`Danger`) — never hardcoded.
- **Color consistency audit (theme-purity invariant).** There must be **no hardcoded color anywhere outside `theme.go`** — no hex, no `lipgloss.Color("<n>")`. Every styled element references the theme via the derived `styles.*` vars/styles. Common leak spots and the rule: list glyphs must be colored (`styles.IconOnline`/`IconOffline`), not left raw (a raw glyph inherits the terminal's default fg and silently ignores the theme); both ping graphs read theme colors; modal content uses the `styles.Modal*` surface styles. To verify after a change: `grep -rnE 'lipgloss\.Color\("(#|[0-9])' internal/` must return nothing outside `theme.go`.
- **Opaque modals (no bleed) with tonal depth.** Overlays are fully opaque rectangles filled with the elevated `Surface` color (`#1a211b`, a tonal layer above the `Background`), giving the "floating" depth the style guide asks for. Every content span uses a `styles.Modal*` style that bakes in `Surface`; every line is padded full width via `modalLine`/`styles.ModalFill` (and `modalRow` for two-column rows, which surface-fills the gap); the container sets explicit `Width`/`Height` **including** padding plus `Background`+`BorderBackground` so the whole bounding box overwrites the view behind it. A sharp `NormalBorder` (Primary) frames it, with the title as a `⌐ TITLE ¬` tab bar. `overlayCenter` splices it over the base. The routes overlay is a `DESTINATION/GATEWAY/LATENCY/STATUS` table with `StatusOK`/`StatusWarn`/`StatusErr` chips; the help overlay is grouped key tables. (Verify opacity after edits: render the overlay over an all-`X` background and assert no `X` survives inside the border bounds.)

Before changing any interaction, consult the keybinding matrix and overlay specs in `design-spec.md` — keys are scoped global vs. context (list/modal) and must not collide.

## Keybindings (implemented)

Keys are **scoped**. In the **main view** the global hotkeys below apply. While **any overlay is open**, `Update` routes every key to `updateOverlay` (the global hotkeys do **not** fire) — see the modal-specific table — and the only thing that crosses back to the main view is closing the overlay. `ctrl+c` always quits.

### Global hotkeys (main view)

| Key | Action |
| :-- | :-- |
| `j`/`k`, ↑/↓ | Move selection in the peer list (wraps around at top/bottom) |
| `/` | Open search (Input Mode); type to fuzzy-filter (hostname + tags). `↑↓`/`Ctrl+j`/`Ctrl+k` navigate while typing; `Enter`/`Esc` apply (blur); `Esc` again (in list) clears |
| `c` | Toggle the local tailnet connection — runs `tailscale up`/`down` interactively (shows auth URLs); button + footer reflect live state |
| `x` | Toggle highlighted peer as the active exit node — executes `tailscale set --exit-node=…` (only if it offers exit; off if already active). Footer hint `[x] Exit Node` shows only on exit-capable peers |
| `e` | Open the **Routes** overlay (`stateRoutes`) — only on a peer with advertised routes |
| `l` | Open the **Accounts** modal (`stateAccounts`, live profiles) |
| `S` | Open the **Advanced Settings** modal (`stateSettings`; uppercase / shift+s) |
| `R` | Open the **Routing Management** modal (`stateRouting`; uppercase / shift+r) |
| `v` | Open/close the **Logs** overlay (`stateLogs`) |
| `O` | Operator setup — suspends the TUI and runs interactive `sudo tailscale set --operator=$USER` |
| `?` | Open/close the **Help** overlay (`stateHelp`) |
| `esc` | In the main view: clear an applied search filter (otherwise no-op) |
| `q` | Quit the app (ignored while the search box is focused / filtering) |
| `s` / `r` | **Reserved no-ops** — explicitly ignored, held for the future SSH-as-action (`s`) and a routing action (`r`); never fall through to the list |
| `ctrl+c` | Quit (always, even mid-search or with an overlay open) |

### Modal-specific hotkeys

Inside an overlay, `esc`/`q` close it (except the Routing CIDR editor — see below). Other keys are scoped to that modal only:

| Modal (state) | Keys |
| :-- | :-- |
| Help (`stateHelp`) | `j/k`·arrows·page scroll the viewport · `?`/`esc`/`q` close |
| Routes (`stateRoutes`) | `j/k`·arrows·page scroll the viewport · `esc`/`q` close |
| Logs (`stateLogs`) | `j/k`·arrows·page scroll the viewport · `v`/`esc`/`q` close |
| Accounts (`stateAccounts`) | `j/k` navigate · `Enter` switch to highlighted (inactive) profile (`sudo tailscale switch …` via `tea.ExecProcess`) · `a` add (`sudo tailscale login` via `tea.ExecProcess`, interactive for the auth URL) · `d` remove highlighted (inactive) profile (`sudo tailscale switch remove …`) · `l` logout (`sudo tailscale logout`) · `esc`/`q` close. All four mutating actions close the modal (`state = stateMain`) before dispatch; when the profile store is locked the modal renders the "run with sudo" hint and `accountsLockedMsg` polling is log-suppressed |
| Advanced Settings (`stateSettings`) | `j/k` navigate the 5 toggles · `Space` toggle (`tailscale set --<flag>=<bool>`, optimistic + reconcile) · `esc`/`q` close |
| Routing — **list mode** (`stateRouting`) | `j/k` navigate (toggle + routes) · `Space` toggle exit-node advertise (item 0 only) · `a` enter CIDR input mode · `d` remove highlighted route (cursor clamps) · `Enter` open the **Command Room** confirmation · `esc`/`q` close. Edits stay local until applied in the Command Room |
| Routing — **CIDR input mode** (`stateRouting`, `routingInputMode`) | type to edit the `bubbles/textinput` · `Enter` confirm (validated by `net.ParseCIDR`; invalid → red flash + clear, stays in input mode) · `Esc` **cancel** the entry (returns to list mode — does **not** close the modal). Input mode owns every key, so `q`/`esc` can't close the modal mid-entry. The field is **pre-filled with the last `[d]`-deleted route** (pseudo-undo) with the cursor at the end |
| Routing — **Command Room** (`stateRoutingConfirm`) | `Enter` **APPLY** — runs `tailscale set --advertise-…` async, closes the modal, refreshes status+prefs · `c`/`C` **COPY** the command to the clipboard (flashes `✓ Copied!`, stays put) · `Esc`/`q` **BACK** to the routing list without applying |

Keys from the design spec not yet wired: `p` (ping-as-action) and `s` (SSH-as-action — lowercase `s` is reserved and currently a no-op; Advanced Settings moved to uppercase `S`).
